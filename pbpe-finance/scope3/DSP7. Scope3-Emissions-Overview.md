Scope 3について、詳しく説明してください。

## Scope 3 排出量とは何か — 基礎から PBPE による変革まで

---

## 1. Scope 3 の基本定義（GHGプロトコル）

温室効果ガス（GHG）プロトコルは、企業の排出量を以下の3つに分類します。

| スコープ | 定義 | 例 |
|----------|------|------|
| **Scope 1** | 自社による**直接排出** | 工場のボイラー、社用車の燃料 |
| **Scope 2** | 購入した**電力・熱・蒸気**の間接排出 | 電気使用に伴う発電側の排出 |
| **Scope 3** | **バリューチェーン全体**のその他の間接排出 | 原材料調達、輸送、製品使用、廃棄 |

### Scope 3 はさらに15のカテゴリに分類される

| カテゴリ | 上流／下流 | 具体例 |
|----------|------------|--------|
| 1. 購入した製品・サービス | 上流 | 原材料の製造 |
| 2. 資本財 | 上流 | 機械設備の製造 |
| 3. 燃料・エネルギー関連活動 | 上流 | 燃料の採掘・精製 |
| 4. 上流の輸送・配送 | 上流 | サプライヤーから工場までの輸送 |
| 5. 事業から出る廃棄物 | 上流 | 製造工程の廃棄物処理 |
| 6. 出張 | 上流 | 社員の飛行機移動 |
| 7. 雇用者の通勤 | 上流 | 社員の車通勤 |
| 8. 上流のリース資産 | 上流 | 賃貸倉庫の運用 |
| **9. 下流の輸送・配送** | 下流 | 製品を顧客へ配送 |
| **10. 販売した製品の加工** | 下流 | 製品を使った製造工程 |
| **11. 販売した製品の使用** | 下流 | 製品の稼働時のエネルギー消費 |
| **12. 販売した製品の廃棄** | 下流 | 製品のリサイクル・埋立 |
| **13. 下流のリース資産** | 下流 | リース先での運用 |
| **14. フランチャイズ** | 下流 | フランチャイズ店の運営 |
| **15. 投資** | 下流 | 投資先企業の排出 |

## 2. なぜ Scope 3 が「世界最大の課題」なのか

### ① Scope 3 が排出量の大部分を占める

| 業種 | Scope 3 の割合（全排出量に占める） |
|------|-------------------------------------|
| 小売 | 90% 以上 |
| 食品・飲料 | 80–95% |
| 自動車 | 70–90% |
| 建設 | 80% 以上 |
| 化学 | 60–80% |
| IT・サービス | 50–70% |

→ **多くの企業にとって、Scope 1+2 は氷山の一角。Scope 3 が本体。**

### ② 現状の Scope 3 算定は「推定」でしかない

現在の一般的な手法：

- **産業平均値**を使う（例：「牛肉1kg = 27kg CO₂e」という世界平均）
- **金額ベース**で換算（例：「購入額 × 産業別排出原単位」）
- **サプライヤーの自己報告**（信頼性が極めて低い）

**結果**：企業によって同じ製品の排出量が2倍も3倍も違う。

### ③ 二重計上が「構造的に発生」している

最も深刻な問題：

```
例：食品ロス削減プロジェクト
├─ 小売業者：「当社のScope 3削減量として報告」
└─ 食品メーカー：「当社のScope 3削減量として報告」
→ 同じ1トンの削減を2社が同時に主張
```

これは**不正ではなく、現行のルールの抜け穴**。

## 3. PBPE Marketplace による Scope 3 の変革

### 3.1 PBPE の基本フロー：Intent → KPI → Credit → Scope 3

```
┌─────────────────────────────────────────────────────────────────────┐
│  PBPE Scope 3 データフロー                                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ① Impact Intent（INT-ID）                                          │
│  「この農場で肥料を20%削減する」という意図をコード化                 │
│         ▼                                                           │
│  ② KPI 測定（KPI-ID）                                               │
│  AGRIX + MBT55 で実際の削減量を計測（例：150 tCO₂e）                 │
│         ▼                                                           │
│  ③ PBPE Credit 発行（CRD-ID）                                       │
│  削減量を「唯一の価値単位」として発行                                │
│         ▼                                                           │
│  ④ Scope 3 報告（RPT-ID）                                           │
│  企業は「どのCRD-IDを使って報告したか」をRegistryに記録              │
│         ▼                                                           │
│  ⑤ ブロックチェーンに証跡保存                                        │
│  改ざん不可能な形で全履歴を保存                                      │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 二重主張を「構造的に不可能」にする仕組み

**従来**：
- 企業A：「この削減量を報告します」
- 企業B：「同じ削減量を報告します」
- システムは「両方ともOK」→ 二重計上

**PBPE**：
- 企業Aが CRD-4F9A2C1 を使って Scope 3 報告 → Registry に記録
- 企業Bが同じ CRD-4F9A2C1 を使って報告しようとする
- Registry が「この Credit はすでに企業Aによって報告済み」と検知 → **エラーで拒否**

```python
# 疑似コード：Scope 3 二重主張防止
def report_scope3(company_id: str, crd_id: str, amount: float):
    existing = db.query(RegistryEntry).filter(
        kind="scope3_report",
        subject_id=crd_id
    ).first()
    
    if existing and existing.actor != company_id:
        raise DoubleClaimingError(
            f"CRD-{crd_id} already claimed by {existing.actor}"
        )
    
    # 報告を記録...
```

### 3.3 サプライチェーン全体の「削減量の追跡」を実現

**例：ある食品メーカーのScope 3削減**

```
農場A（肥料－30％） → CRD-A01
農場B（土壌炭素＋50t） → CRD-B02
物流会社C（輸送効率化） → CRD-C03
食品メーカーD
    ├─ 購入したCRD-A01（Scope 3 カテゴリ1）
    ├─ 購入したCRD-B02（Scope 3 カテゴリ1）
    └─ 購入したCRD-C03（Scope 3 カテゴリ4＋9）
```

→ **同じ削減量をメーカーと農場が同時に主張できない**  
→ **削減の「所有権」が一意に決まる**

## 4. PBPE が解決する Scope 3 の7つの致命的欠陥

| 現状の問題 | PBPEによる解決 |
|------------|----------------|
| ① データが「推定」でしかない | AGRIX + MBT55 による**実測ベース** |
| ② 二重計上が日常的に発生 | Registryが**一意のCRD-ID**で防止 |
| ③ サプライチェーンが複雑で追えない | 7桁コードで**全プロセスをトレース** |
| ④ 報告コストが莫大（数百万円〜） | 自動連携で **コスト−70%** |
| ⑤ 第三者保証（検証）が高額 | ブロックチェーンで**自動検証** |
| ⑥ 国際的な相互運用性がない | PBPE Registryが**単一の世界台帳**に |
| ⑦ Scope 3 削減が「コスト」でしかない | PBPE Creditとして**収益化可能** |

## 5. 企業の具体的なメリット（定量）

### ある食品メーカー（売上1兆円）の場合

| 項目 | 現状 | PBPE導入後 | 改善 |
|------|------|------------|------|
| Scope 3 算定コスト | 年間2億円 | 年間6,000万円 | **−70%** |
| 第三者検証コスト | 年間5,000万円 | ブロックチェーン自動検証 → ほぼゼロ | **−90%** |

## 6. 政府・国際機関にとっての意義

### 環境省（国内カーボンマーケット）
- J-クレジットとScope 3の連携が可能に
- 「企業のScope 3削減 ＝ 国内の排出削減」として一体的に管理

### 経産省（GX・ESG）
- 日本企業のScope 3報告コストを**業界全体で削減**
- 「日本発のScope 3算定・報告標準」として国際提案可能

### 外務省（気候変動国際交渉）
- UNFCCC（国連気候変動枠組条約）のMRV問題を解決するモデルとして提示
- 途上国のScope 3算定を支援（ODAの効果測定にPBPEを活用）

## 7. まとめ：Scope 3 は PBPE の「キラーアプリケーション」

> **Scope 3 は、世界で最も大きく、最も複雑で、最も二重計上が発生している排出カテゴリ。PBPE Marketplace は、この Scope 3 を「推定から実測へ」「コストから収益へ」「二重計上から一意追跡へ」変える世界で唯一の仕組み。**
