## 1. PBPE Registry（Layer 5）の完全設計

### **1.1 役割**

- **全 PBPE の「公式台帳」**
- 二重計上防止・監査・トレーサビリティの中枢
- 7 桁コード＋ブロックチェーンの“橋渡しレイヤー”

### **1.2 コア概念**

- **RegistryEntry**
    - `id`（RPT-XXXXXXX）
    - `kind`（"credit_issuance" / "credit_retirement" / "trade" / "bond_coupon" / "scope3_report" など）
    - `subject_id`（CRD-ID / BND-ID / TRD-ID / KPI-ID など）
    - `amount`（tCO₂e / PBPE / USD など）
    - `actor`（企業 / 政府 / 個人）
    - `timestamp`
    - `chain_tx_hash`（ブロックチェーン側のハッシュ）

### **1.3 API（イメージ）**

- `POST /registry/record`  
    → 新しい RegistryEntry を作成し、チェーンに書き込む
- `GET /registry/{id}`  
    → 単一エントリの詳細
- `GET /registry/by-subject/{subject_id}`  
    → 特定の PBPE Credit / Bond / Trade の履歴
- `GET /registry/search?actor=...&kind=...`  
    → 監査・ESG レポート用

## 2. ブロックチェーン層（Layer 0）のアーキテクチャ

### **2.1 位置づけ**

- PBPE Registry の **永続・改ざん不可レイヤー**
- フルパブリックチェーンでなくてもよい（最初は permissioned / internal でも可）

### **2.2 最小構成（自前チェーン実装案）**

**ディレクトリ構造案：**

```text
backend/blockchain/
  __init__.py
  block.py   # Block 定義
  chain.py   # Chain 管理（検証・追加）
  ledger.py  # Registry との接続
```

**Block（概念）**

- `index`
- `previous_hash`
- `timestamp`
- `payload`（RegistryEntry の要約＋7桁コード）
- `hash`

**Chain**

- `blocks: List[Block]`
- `add_block(payload) -> Block`
- `is_valid() -> bool`

**Ledger**

- `record_registry_entry(entry) -> chain_tx_hash`
- RegistryEntry を受け取り → Block にしてチェーンに追加 → ハッシュを返す

※ 実際のコードは、後で「ファイル単位」で出せる。

## 4. PBPE Credits の二重計上防止ロジック（実装方針）

### **4.1 原則**

1. **同じ削減量（KPI-ID）は、一度しか PBPE Credit に変換できない**
2. **同じ PBPE Credit（CRD-ID）は、一度しか償却（retire）できない**
3. **同じ削減量を複数企業が同時に主張できない**

### **4.2 ロジック（擬似コードレベル）**

**① 発行時（/credits/issue）**

- 入力：`kpi_id`, `amount_pbpe`, `owner`
- チェック：
    - その `kpi_id` に紐づく `credit_issuance` が Registry に存在しないか？
        - 存在する → **二重発行エラー**
- OK なら：
    - `CRD-ID` を生成（7 桁コード）
    - Registry に `kind="credit_issuance"` で記録
    - ブロックチェーンに書き込み
    - `CRD-ID` を返す

**② 償却時（/credits/retire）**

- 入力：`crd_id`, `amount`, `actor`
- チェック：
    - その `crd_id` の発行量以上がすでに retire されていないか？
        - 超える → **二重償却エラー**
- OK なら：
    - Registry に `kind="credit_retirement"` を記録
    - ブロックチェーンに書き込み

**③ Scope 3 報告時（/scope3/report）**

- 入力：`company_id`, `crd_id`, `amount`
- チェック：
    - その `crd_id` がすでに他社の Scope 3 報告に使われていないか？
        - 使われている → **二重主張エラー**
- OK なら：
    - Registry に `kind="scope3_report"` を記録
    - ブロックチェーンに書き込み

## まとめ

- **Registry 設計**で「何を記録するか」を固定し
- **Blockchain 層**で「改ざん不可の履歴」を持たせ

- **二重計上防止ロジック**で「同じものを二度使えない」制約をコード化する

> **“二重計上ゼロの農業 × 気候金融 × ESG OS”**

として、技術的にも論理的にも一貫した形になる。
